# Git 컨벤션

배달 주문 관리 플랫폼 팀 프로젝트의 Git 협업 규칙입니다. 6명이 하나의 모노레포에서 작업하므로 아래 규칙을 지켜주세요.

---

## 1. 브랜치 전략

```
main       
     ├─ feat/user       (User/Auth 담당)
     ├─ feat/store      (Store/카테고리/지역 담당)
     ├─ feat/product    (Product/AI 담당)
     ├─ feat/order      (Order 담당)
     ├─ feat/payment    (Payment 담당)
     ├─ 나머지 알아서
     └─ feat/common     (공통 인프라 담당)
```

**규칙**
- 기능 개발은 `feat/{도메인}` 브랜치에서 진행
- 세부 작업으로 더 쪼갤 경우: `feat/product-ai`, `feat/product-search` 처럼 하이픈으로 구분

---

## 2. 커밋 메시지 컨벤션

**형식**

```
[{type}]: {제목}

{본문}
```

**예시**

```
feat: 상품 등록 API 구현
fix: 상품 검색 시 페이징 오류 수정
docs: API 명세서 업데이트
```

**Type 종류**

| type       | 설명 |
|------------|---|
| `Feat`     | 새로운 기능 추가 |
| `Fix`      | 버그 수정 |
| `Docs`     | 문서 수정 (README, 명세서 등) |
| `Style`    | 코드 포맷팅, 세미콜론 등 (기능 변화 없음) |
| `Refactor` | 리팩토링 (기능 변화 없이 코드 개선) |
| `Test`     | 테스트 코드 추가/수정 |
| `Chore`    | 빌드 설정, 패키지 설치, 초기 세팅 등 |

**규칙**
- 제목은 한글로 간결하게, 마침표 없이
- 제목은 명사형으로 끝내기
- 여러 기능을 한 커밋에 섞지 않기

---

## 3. Pull Request 규칙

**PR 제목**: 커밋 컨벤션과 동일한 형식 사용
```
[Feat]: 상품 도메인 CRUD 및 검색 기능 구현
```

**PR 본문에 포함할 내용**
- 작업 내용 요약 (뭐 구현/수정했는지)
- 리뷰어가 중점적으로 봐야 할 부분
- 테스트 방법 또는 확인 사항

**PR 템플릿 **

```markdown
## 작업 내용
- 

## 리뷰 포인트
- 

```

**규칙**
- **작은 단위로 자주 PR** — 한 번에 너무 많은 코드를 올리면 리뷰가 어려움
- 최소 **1명 이상의 승인(Approve)** 후 병합
- 병합은 PR 작성자 본인이 (리뷰 반영 확인 후)
- Merge 방식은 팀에서 통일 (`Squash and merge`??)

---

## 4. 코드 리뷰 규칙

- 리뷰는 **빠르고 간결하게** — 하루 안에 확인하는 것을 목표로
- 문제점은 감정 없이 **건설적으로** 제안
- 단순 제안(nit)과 반드시 고쳐야 할 것(must)을 구분해서 코멘트
- 리뷰어가 approve하면 작성자가 병합

---

## 5. 충돌(Conflict) 방지 및 해결

- 작업 전 항상 `git pull origin main`로 최신화
- 하루 작업 시작 시, 작업 중간에도 주기적으로 main 변경사항을 자기 브랜치에 반영

```bash
git checkout feature/product
git pull origin main
```

- 충돌 발생 시 혼자 해결하지 말고, 관련 코드 작성자와 함께 확인

---

## 6. .gitignore 필수 항목

아래는 반드시 Git에서 제외 (커밋 금지):

- 빌드 결과물: `build/`, `.gradle/`
- IDE 설정: `.idea/`, `*.iml`
- **민감정보**: DB 비밀번호, JWT 시크릿, **Gemini API 키** 등
    - `application-secret.yml` 또는 환경변수로 분리하고 gitignore에 추가
    - API 키가 올라가면 요금 폭탄/키 정지 위험이 있으니 각별히 주의

---

## 7. 절대 하지 말 것!!

- `main`에 직접 push
- 리뷰 없이 병합
- API 키·비밀번호 커밋
- 남의 feature 브랜치를 임의로 수정
- 하나의 거대한 PR로 몰아서 올리기
